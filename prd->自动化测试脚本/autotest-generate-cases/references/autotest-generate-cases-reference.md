# 基于需求生成文本测试用例 — 参考手册

> **本文件是测试用例参考手册，不是执行指令，AI 生成用例时不主动读取此文件。**
> 只有在执行协议中明确指引读取本测试用例参考手册的某章节时，才按需读取对应章节。
> 执行协议路径：`../SKILL.md`

## 需求信息提取

写用例前，先从需求中提取：

- 功能点编号和名称。
- 页面名称和入口路径。
- 用户角色和权限（每种角色的能力边界）。
- 适用端（iOS、Android、HarmonyOS、Web 等）。
- 前置条件和前置数据状态。
- **所有状态矩阵/状态表**（需求中以表格形式列出的状态、条件与对应表现，必须逐行展开为独立用例）。
- 按钮文案、弹窗文案、Toast 文案、空态/缺省文案（每个文案都要有对应展示验证用例）。
- 成功结果（UI层面的表现，不是数据层）。
- 失败和异常结果（每种异常分别列出）。
- 边界值和输入限制（最大值、最小值、临界值分别列出）。
- **网络依赖**：哪些功能依赖网络？无网络/弱网/网络恢复时的表现是什么？需求是否明确写了网络异常的处理逻辑？
- **前后台/进程状态**：哪些功能有持续性行为？切后台、锁屏、杀进程后的行为是什么？需求是否明确写了"后台不中断""退出不中断"等声明？
- 与旧版本相比新增、修改、删除的行为。
- 版本差异（新服务端 vs 旧服务端、新客户端 vs 旧客户端、不同硬件/固件版本等）。
- **外部依赖**：哪些功能依赖外部组件（硬件、第三方服务、蓝牙、Wi-Fi等）？每种依赖断开时的异常场景。
- 待确认问题。

如果需求没有编号，可以临时用功能点名称关联用例。

## 用例拆分原则

### 核心原则：一个验证点 = 一条用例

**绝不允许**把多个验证点合并为一条"综合用例"。以下是对拆分规则的具体说明：

### 1. 按前置状态拆分

不同的前置状态/条件导致不同的UI展示或交互逻辑时，必须拆成多条用例。

❌ 错误示例（合并）：
```
用例：修改属性-正向流程
步骤：1. 点击属性名称 2. 观察弹窗 3. 修改内容 4. 勾选批量修改 5. 点击确定
预期：弹窗标题正确、输入框预置正确、修改成功、批量修改生效
```

✅ 正确示例（拆分）：
```
修改属性-按钮显示（验证按钮是否出现）
修改属性-弹窗展示（验证弹窗标题、布局、字段）
修改属性-输入框预置（验证预置文案）
修改属性-手动修改单条（验证修改单条数据）
修改属性-从列表选人（验证从列表选择后的回显）
修改属性-选择历史记录（验证快选列表）
修改属性-清空已选（验证清空操作）
修改属性-同步修改所有-不选（验证不勾选只改当前）
修改属性-同步修改所有-选中（验证勾选批量修改）
修改属性-多次修改（验证修改后再修改）
修改属性-历史记录列表（验证已修改项出现在快选列表）
```

### 2. 按每种异常/状态组合拆分

需求中的状态矩阵，每一行必须展开为一条独立用例。

❌ 错误示例（合并提及）：
```
用例：删除操作-各种状态
预期：处理中提示"正在处理中"、运行中提示"正在运行中"、未连接按钮置灰
```

✅ 正确示例（逐条拆分）：
```
删除操作-已连接（正常删除成功）
删除操作-已连接-运行中（不允许删除，Toast提示）
删除操作-已连接-处理中（不允许删除，Toast提示）
删除操作-已连接-有未完成数据（允许删除）
删除操作-未连接（按钮不可点击/置灰）
```

### 3. 按时间/数值区间拆分

有格式化规则的展示字段，每个区间必须拆成独立用例。

❌ 错误示例（合并）：
```
用例：时间展示规则验证
预期：60分钟前显示分钟前、超过60分钟显示小时前、超过7天显示日期
```

✅ 正确示例（逐区间拆分）：
```
时间展示-1h以内（展示x分钟前）
时间展示-1h以上24h内（展示x小时前）
时间展示-1天以上7天内（展示x天前）
时间展示-7天以上（展示完整日期时间格式）
时间展示-取值规则（确认取哪个时间字段）
```

### 4. 按操作分支拆分

一个功能有多个操作路径（如弹窗的取消和确认、删除的确认和取消），每个分支必须拆成独立用例。

❌ 错误示例（合并）：
```
用例：导出功能
步骤：点击导出 > 弹窗选择格式 > 确认导出 / 取消
预期：未生成时Toast提示、已生成时弹窗选择、确认后导出成功、取消后关闭弹窗
```

✅ 正确示例（逐分支拆分）：
```
导出内容-未生成（Toast提示"请先生成"）
导出内容-已生成（弹出选择格式弹窗）
导出内容-弹窗（验证弹窗标题、选项、默认选中、按钮）
导出内容-弹窗-取消（弹窗关闭回到菜单）
导出内容-弹窗-确认（loading后跳转结果页）
导出内容-弹窗-选择格式A（选择不同格式的行为）
导出内容-弹窗-选择格式B（选择不同格式的行为）
导出内容-文件命名规则（验证命名格式）
```

### 5. 按角色视角拆分

涉及多角色交互的功能，按角色视角和交互路径拆分。

❌ 错误示例（合并）：
```
用例：分享功能验证
预期：分享成功、被分享者可查看、外部用户不可查看
```

✅ 正确示例（逐角色逐路径拆分）：
```
分享-私信渠道（选私信对话发送）
分享-群组渠道（选群组对话发送）
分享-消息类型（验证消息卡片内容）
分享-查看详情-被分享者（被分享者点击后的行为）
分享-查看详情-分享者未生成内容时（缺省页展示）
分享-被分享者不能执行高级操作
分享-消息点击后-分享者（权限列表更新）
分享-消息点击后-被分享者（列表刷新）
分享-多次分享（同一用户可获得新权限）
分享-无权限（被移除后）
分享-外部用户（不可查看）
权限管理-添加成员
权限管理-移除成员
权限管理-移除后再分享可获权
```

### 6. 按版本/外部依赖差异拆分

需求提到新/旧版本差异、不同外部依赖差异时，必须按版本/依赖分别写用例。

示例（仅为说明格式）：
```
功能入口-新版本（新客户端+新服务端的表现）
功能入口-旧版本（新客户端+旧服务端的表现）
功能重连-依赖版本A（版本A的恢复行为）
功能重连-依赖版本B（版本B的恢复行为）
```

### 7. UI展示类用例必须单独生成

以下场景必须生成独立的展示验证用例，不能只写在操作类用例的预期结果里：

- 页面缺省/空态文案和布局
- 列表展示字段（每个字段单独验证）
- 排序规则
- 名称截断/省略号规则
- 时间/数值格式化规则
- 图标颜色/样式/形状规则
- 进度条/百分比/数值展示格式
- 弹窗标题、按钮文案、描述文案
- Tab名称和选中样式
- 状态标签文案
- 视觉指示效果（指示灯、标记、闪烁等）

示例（仅为说明格式）：
```
列表页-展示字段（验证每个字段是否出现且内容正确）
列表页-排序（验证排序规则）
列表页-名称截断（验证名称至多N行，超出省略号）
列表页-缺省页-无数据（验证空态文案和布局）
列表页-缺省页-另一Tab无数据（验证不同Tab的不同缺省文案）
状态图标-状态A（验证图标颜色/形状/跳转目标）
状态图标-状态B（验证图标颜色/形状/跳转目标）
数值展示-格式（验证展示格式如进度+%、xx/总容量等）
```

## 覆盖思路

### 必须覆盖的12种类型

1. **正向流程**：用户按主路径完成操作。
2. **UI展示验证**：每个页面元素、文案、格式、排序、图标样式都要有独立展示验证用例。这是最容易被漏掉的类型。
3. **权限场景**：有权限、无权限、不同角色，按角色视角拆分。**安全场景**（越权访问、水平越权、垂直越权）也归入此类。
4. **状态组合**：需求中的状态矩阵每行一条用例；同一功能在不同前置状态下的表现分别验证。
5. **输入边界**：为空、最小值、最大值、临界值、超过限制、非法字符、前后空格，每种单独一条。
6. **异常流程**：取消、返回、关闭弹窗、重复操作、服务端失败、外部依赖断开（根据项目实际依赖列出，如硬件/蓝牙/第三方服务等），每种单独一条。
7. **网络状态**：不同网络条件下功能的表现，每种网络状态单独一条用例。（详见下方"网络状态覆盖"章节）
8. **前后台/进程状态**：App在不同前后台/进程状态下功能的表现；Web端的会话/浏览器标签页状态；每种状态单独一条用例。（详见下方"前后台/进程状态覆盖"章节）
9. **版本兼容**：新/旧服务端差异、新/旧客户端差异、不同依赖版本差异、跨端行为一致性、不同浏览器兼容性。
10. **数据一致性**：同一数据在不同页面、不同端、不同时间点的展示是否一致；修改后关联页面是否同步刷新；离线操作后上线数据是否正确合并。
11. **并发/竞态**：多人同时操作同一资源、快速连续点击（防抖/防重）、离线操作后上线冲突、同一用户多端同时操作。
12. **回归**：修改已有功能时旧行为必须回归验证；配套新功能上线时关联旧功能必须回归；版本升级时已有功能必须回归。

### 网络状态覆盖

所有涉及网络请求的功能（无论是App、Web还是API），必须覆盖以下网络场景，每种单独一条用例：

| 网络状态 | 说明 | 适用平台 | 判断哪些功能需要覆盖 |
|---------|------|---------|----------------|
| Wi-Fi正常 | 主路径默认条件 | App/Web | 所有涉及网络请求的功能 |
| 移动网络（4G/5G）正常 | 移动网络条件 | App | Wi-Fi和移动网络可能有不同表现的功能 |
| 无网络（断开所有网络） | 完全无网络时的表现 | App/Web/API | 所有涉及网络请求的功能 |
| 网络弱（高延迟/低带宽） | 网络慢但不完全断开 | App/Web | 对实时性要求高的功能 |
| 网络恢复 | 从无网络恢复到正常 | App/Web | 需求中声明了"自动重试"或"恢复后继续"的功能 |
| 局域网特殊场景 | 连接特定网络时外网不可用 | App | 需要连接设备热点/局域网的功能 |
| 请求超时 | 服务端响应超时 | Web/API | 所有涉及网络请求的功能（需确认超时时间和重试策略） |

用例命名示例（仅为说明格式）：
```
数据提交-无网络（提交失败，下次进入重试）
数据提交-网络恢复（恢复后自动重试成功）
实时处理-网络异常（弹窗提示网络异常）
实时处理-网络恢复（恢复后自动继续）
数据导出-无网络（导出失败，显示网络异常提示）
文件下载-无网络（下载失败，Toast提示）
```

**重要原则**：
- **按功能×网络状态逐条覆盖**，不能用一条"所有功能无网络验证"合并所有功能。不同功能在无网络下的UI表现和恢复逻辑不同，必须写成独立用例。
- **判断哪些功能需要覆盖网络状态**：看该功能是否依赖网络请求。如果功能纯粹本地操作，不需要覆盖网络状态。

### 前后台/进程状态覆盖

#### App端

| App状态 | 说明 | 判断哪些功能需要覆盖 |
|---------|------|----------------|
| 切到后台（Home键/手势） | App仍在运行但不在前台 | 有持续性行为的功能 |
| 切回前台 | 从后台回到App | 有持续性行为的功能（验证状态是否正确恢复） |
| 锁屏 | 手机锁屏 | 有持续性行为的功能 |
| 杀进程（强杀App） | App进程被终止 | 有持续性行为或需要恢复的功能（进程被杀后内存全部丢失，恢复逻辑与后台运行完全不同） |
| 音频/通话通道被占用 | 其他App占用音频 | 有音频播放的功能 |
| 退出当前功能页面（返回上一级） | 在App内返回 | 正在执行持续性操作的功能 |
| 跨App返回（从其他App切回） | 去了别的App再回来 | 有实时数据刷新需求的功能 |

#### Web端

| Web状态 | 说明 | 判断哪些功能需要覆盖 |
|---------|------|----------------|
| 切换浏览器标签页 | 用户切换到其他标签页 | 有持续性行为的功能 |
| 切回浏览器标签页 | 用户切回当前标签页 | 验证页面数据是否自动刷新/状态是否恢复 |
| 关闭浏览器标签页 | 用户关闭当前页面 | 正在执行持续性操作的功能 |
| 浏览器刷新（F5） | 用户手动刷新页面 | 页面状态是否丢失、表单是否保留、操作是否中断 |
| 会话超时/登录态过期 | Token过期或Session失效 | 所有涉及鉴权的功能 |
| 前进/后退导航 | 用户使用浏览器前进后退 | 表单数据是否丢失、页面状态是否正确恢复 |

用例命名示例（仅为说明格式）：
```
持续操作-退出当前页面（操作是否中断）
持续操作-切后台（操作是否中断）
持续操作-锁屏（操作是否中断）
持续操作-杀进程（重新进入后状态）
持续操作-通道被占用（操作是否暂停）
持续操作-自然结束（结束后的状态）
持续操作-切回前台（状态恢复）
实时操作-切后台（不中断）
实时操作-进程杀死（无法恢复，重进可查看已有数据）
```

**重要原则**：
- **按功能×状态逐条覆盖**，不能用一条合并所有功能。
- **需求明确标注了"后台不中断"或"退出不中断"的功能**，必须验证这些声明是否成立，分别写"切到后台"和"切回前台"两条用例。
- **杀进程是最极端的场景**，必须单独验证。
- **判断哪些功能需要覆盖前后台状态**：看该功能是否有持续性行为。纯展示页面或一次性操作通常不需要覆盖。

### 特定平台的额外覆盖规则

根据项目的适用平台，选择对应的额外覆盖规则：

#### Web后台管理系统

| 覆盖类型 | 说明 | 用例示例 |
|---------|------|---------|
| 浏览器兼容性 | 不同浏览器下的表现差异 | 列表页-Chrome展示、列表页-Safari展示 |
| 响应式布局 | 不同屏幕宽度下的布局适配 | 列表页-宽屏布局、列表页-窄屏布局 |
| 表格分页/筛选/排序 | 数据表格的交互操作 | 数据列表-分页切换、数据列表-筛选组合 |
| 批量操作 | 多选后的批量行为 | 数据列表-批量删除-部分失败、数据列表-批量导出 |
| 表格数据量大 | 大数据量下的渲染和性能 | 数据列表-10000条数据加载 |
| 会话超时 | 长时间未操作后登录态过期 | 编辑表单-会话超时提交 |
| CSRF/XSS安全 | 安全性基础验证 | 搜索框-XSS注入、表单提交-CSRF验证 |
| 数据导出 | 不同格式的导出行为 | 导出-CSV、导出-Excel、导出-大数据量 |
| 并发操作 | 同一数据被多人同时操作 | 编辑-并发修改同一数据 |
| 操作审计日志 | 关键操作是否有日志记录 | 删除-审计日志记录 |

#### 纯API接口

| 覆盖类型 | 说明 | 用例示例 |
|---------|------|---------|
| 参数校验 | 必填/可选、类型、枚举值、长度限制 | POST /api/xxx-缺少必填字段、POST /api/xxx-字段类型错误 |
| 鉴权/鉴权失败 | 无Token、过期Token、错误Token | GET /api/xxx-无Token、GET /api/xxx-过期Token |
| 幂等性 | 重复请求的结果是否一致 | POST /api/xxx-重复提交 |
| 并发请求 | 高并发下的响应和数据一致性 | GET /api/xxx-100并发请求 |
| 超时处理 | 请求超时的错误码和重试策略 | GET /api/xxx-服务端超时 |
| 分页/排序/过滤 | 参数组合的正确性 | GET /api/xxx-page=0&size=100&sort=name |
| 字段脱敏 | 敏感字段是否正确脱敏 | GET /api/xxx-手机号脱敏 |
| 响应格式 | 返回数据结构和状态码 | POST /api/xxx-成功201、POST /api/xxx-失败400/409/500 |
| 版本兼容 | 不同API版本的行为差异 | v1接口-vs-v2接口 |
| 限流/防刷 | 超出调用频率限制的表现 | GET /api/xxx-超出限流（429） |

#### 桌面应用

### 组合功能点的数量计算

如果一个功能点同时包含多种类型的交互行为（如章节功能 = 展示 + 生成 + 状态管理 + 跳转），不应按单一类型套标准，而是按各子行为的类型分别查表得到参考条数后 **叠加**。

示例：
- 章节内容展示（页面展示类）：8条
- 竑节生成（持续性操作类）：10条
- 竑节跳转（正向+异常）：5条
- 竑节分享（权限类）：4条
→ 参考合计：27条

### 关于总用例数量

**不再使用总公式控制用例数量。** 用例数量的质量保障完全依靠「输出前质量门槛」中的覆盖类型检查清单（12种覆盖类型是否齐全、是否合并验证点、是否逐行展开状态矩阵等）。这些检查项有明确的逻辑支撑，比任何总数公式更可靠。

如果某个功能点的实际用例数明显低于其类型的参考范围，应回头检查是否合并了多个验证点或遗漏了覆盖类型。但参考范围本身不是硬性门槛——如果功能点确实简单（如一个只有3行的状态矩阵），N+3条就够了，不需要强行凑到8条。

### 处理方式标签

- 🔴 阻塞：不确认就无法生成可靠用例。

### 需求质量问题标签

- 🔴 矛盾：同一功能在不同位置描述不一致。
- 🟡 含糊：描述无法确定唯一的预期行为。
- 🔴 缺失：关键场景/异常/边界/文案未提及。
- 🟡 不可测：描述无法在UI层面客观验证。

- ⚪ 超范围：描述了技术实现细节而非用户行为。

## PRD 输入格式适配

本 Skill 支持以下格式的 PRD 输入，每种格式的处理方式不同：

| 格式 | 处理方式 | 说明 |
|------|---------|------|
| **Markdown (.md)** | 直接读取 | 最理想的输入格式，结构清晰，易于提取功能点、表格、文案 |
| **PDF** | 需先提取为文本 | 使用 Python 的 `pdfplumber` 或 `PyPDF2` 提取全文；如果 PDF 包含复杂表格或图片，提取可能不完整，需标注"原文可能丢失格式信息" |
| **Word (.docx)** | 需先提取为文本 | 使用 Python 的 `python-docx` 提取段落和表格；表格数据可能需要手动校验 |
| **飞书/钉钉文档** | 用户粘贴文本 | 用户从在线文档复制文本直接粘贴到对话中；格式可能丢失，需标注"粘贴格式可能与原文有差异" |
| **口述需求整理稿** | 用户直接提供 | 由用户自行整理的结构化文本；格式不规范时需标注"非标准需求文档格式" |

   # 仅供提取文本参考，不是测试用例的内容

## 下游衔接准备

虽然本 Skill 生成的是文本测试用例（不是自动化脚本），但输出的格式和内容需要方便下游"测试用例→自动化脚本" Skill 消费。

### 衔接要求

1. **页面名称稳定且前后一致**：同一个页面在所有用例中使用相同的名称（不要一会儿写"详情页"，一会儿写"信息页"）。
2. **入口路径明确**：每条用例的入口路径从起始页面开始完整描述，下游可根据入口路径编写归位脚本。
3. **业务元素名称稳定**：按钮、菜单、Tab、弹窗等使用一致的名称（如始终写"右上角更多"，不要一会儿写"更多"一会儿写"三点菜单"）。
4. **预期文案精确**：Toast、弹窗标题、按钮文案尽量使用需求原文，标注"文案待确认"的需后续补齐。
5. **测试数据明确**：测试数据要写清楚具体值（如"录音名称：测试录音_001"），不要只写"一条测试数据"。
6. **可观测校验点明确**：每条用例的可观测校验点要能在 UI 层面客观判断。
7. **页面路径约定 Sheet**：这个 Sheet 是下游自动化脚本编写归位逻辑的关键依据。
8. **公共前置条件 Sheet**：这个 Sheet 是下游自动化脚本编写 fixture/setup 的关键依据。
9. **自动化建议列**：标注了"适合自动化"的用例，是下游优先自动化的候选。

### 衔接边界

- **本 Skill 的输出边界**：文本测试用例 Excel 表格（7 个 Sheet），不含任何脚本代码、locator 定位符、API 路径。
- **下游 Skill 的输入边界**：本 Skill 的 Excel + 页面 UI 树 JSON + 工程框架代码。
- **中间不缺失的信息**：页面名称 → UI 树映射、业务元素名称 → locator 映射，由下游 Skill 根据 UI 树完成，不需要在本 Skill 的用例中预填。

## Excel 生成机制

本 Skill 的最终产物是 Excel 表格。Claude Code 无法直接写 `.xlsx` 文件，必须通过 Python 的 `openpyxl` 库生成。

### 生成方式

在工作流程完成后，使用以下 Python 脚本生成 Excel 文件。脚本中的数据来自前面步骤生成的 Markdown 内容，由 AI 将 Markdown 结构化数据填入脚本中的列表变量，然后执行脚本输出 xlsx 文件。

```python
# ============================================================
# Excel 生成脚本 —— 仅供将 Markdown 用例数据转为 xlsx 文件使用
# 这不是测试用例的内容，而是工具性代码。
# ============================================================

import openpyxl
from openpyxl.styles import Font, Alignment, PatternFill, Border, Side

def create_testcase_excel(output_path, data):
    """
    data 是一个字典，包含 7 个 Sheet 的数据。
    结构如下：
    {
        "说明": [{"项目": "...", "内容": "..."}],
        "公共前置条件": [{"编号": "...", "分类": "...", "前置条件/测试数据": "...", "备注": "..."}],
        "页面路径约定": [{"页面/场景": "...", "标准入口路径": "...", "脚本生成说明": "..."}],
        "需求质量问题": [{"序号": "...", "问题描述": "...", ...}],
        "需求覆盖矩阵": [{"功能点编号": "...", "功能点名称": "...", ...}],
        "待确认问题": [{"序号": "...", "问题": "...", ...}],
        "测试用例": [{"用例ID": "...", "用例名称": "...", ...}],
    }
    """
    wb = openpyxl.Workbook()

    header_font = Font(bold=True, size=11)
    header_fill = PatternFill(start_color="4472C4", end_color="4472C4", fill_type="solid")
    header_font_white = Font(bold=True, size=11, color="FFFFFF")
    thin_border = Border(
        left=Side(style='thin'), right=Side(style='thin'),
        top=Side(style='thin'), bottom=Side(style='thin')
    )
    wrap_alignment = Alignment(wrap_text=True, vertical='top')

    # 各 Sheet 的列定义
    sheet_configs = {
        "说明": {
            "columns": ["项目", "内容"],
            "data_key": "说明"
        },
        "公共前置条件": {
            "columns": ["编号", "分类", "前置条件/测试数据", "备注"],
            "data_key": "公共前置条件"
        },
        "页面路径约定": {
            "columns": ["页面/场景", "标准入口路径", "脚本生成说明"],
            "data_key": "页面路径约定"
        },
        "需求质量问题": {
            "columns": ["序号", "问题描述", "关联功能点", "问题类型", "影响范围", "原文出处", "建议确认人", "处理建议"],
            "data_key": "需求质量问题"
        },
        "需求覆盖矩阵": {
            "columns": ["功能点编号", "功能点名称", "覆盖用例数", "覆盖用例ID范围", "覆盖状态", "优先级", "需求问题", "备注"],
            "data_key": "需求覆盖矩阵"
        },
        "待确认问题": {
            "columns": ["序号", "问题", "关联功能点", "影响", "原文出处", "建议确认人", "问题类型标签", "处理方式标签"],
            "data_key": "待确认问题"
        },
        "测试用例": {
            "columns": ["用例ID", "用例名称", "关联功能点", "优先级", "用例类型", "适用端", "入口路径", "前置条件", "测试数据", "操作步骤", "预期结果", "可观测校验点", "数据清理", "自动化建议", "备注"],
            "data_key": "测试用例"
        },
    }

    # 删除默认 Sheet
    default_sheet = wb.active
    wb.remove(default_sheet)

    for sheet_name, config in sheet_configs.items():
        ws = wb.create_sheet(title=sheet_name)
        columns = config["columns"]
        data_key = config["data_key"]

        # 写表头
        for col_idx, col_name in enumerate(columns, 1):
            cell = ws.cell(row=1, column=col_idx, value=col_name)
            cell.font = header_font_white
            cell.fill = header_fill
            cell.border = thin_border
            cell.alignment = Alignment(horizontal='center', vertical='center')

        # 写数据行
        rows = data.get(data_key, [])
        for row_idx, row_data in enumerate(rows, 2):
            for col_idx, col_name in enumerate(columns, 1):
                value = row_data.get(col_name, "")
                cell = ws.cell(row=row_idx, column=col_idx, value=value)
                cell.border = thin_border
                cell.alignment = wrap_alignment

        # 自动列宽（近似）
        for col_idx, col_name in enumerate(columns, 1):
            max_len = len(str(col_name))
            for row in rows:
                cell_val = str(row.get(col_name, ""))
                max_len = max(max_len, min(len(cell_val), 50))
            ws.column_dimensions[openpyxl.utils.get_column_letter(col_idx)].width = max_len + 2

    wb.save(output_path)
    print(f"Excel 文件已生成：{output_path}")
```

### 使用方式

1. AI 先按执行协议的流程生成 Markdown 格式的覆盖矩阵、用例、待确认问题等。
2. AI 将 Markdown 内容结构化为 Python 字典数据（填入 `data` 变量）。
3. 将脚本和数据合并为一个完整的 Python 文件，写入临时路径后执行。
4. 执行后生成 `.xlsx` 文件，告知用户文件路径。

**注意**：
- 这个 Python 脚本是将 Markdown 用例数据转为 xlsx 的工具性代码，**不是测试用例本身**。
- 测试用例的内容（步骤、预期结果等）中禁止出现任何脚本代码。
- 如果用例数据量过大（超过 200 条用例），应分批写入，避免单个脚本过长导致执行失败。
