# 测试用例规范检查 — 执行协议

## 职责边界

本 Skill 只负责检查测试用例文本本身是否规范，并给出文本层自动化候选初判：

- 输入：用户显式提供的测试用例 Excel、测试用例文本，或用户明确提供的同类用例材料。
- 可读取：测试用例工作簿内的用例 Sheet、公共前置条件 Sheet、测试数据说明 Sheet、待确认问题 Sheet 等同一份用例文档内容。
- 输出：输入清单、逐条用例规范审查结果、统一问题清单，写入固定输出目录。
- 不读取工程实现材料：不得主动读取页面映射表、页面 UI 树、截图、PageObject、已有 pytest 用例、正式图片目录、`CLAUDE.md`、`README.md` 或其他工程文件。
- 不判断工程落地条件：不判断元素是否能在 UI 树定位，不判断 PageObject 是否可复用，不判断正式图片是否存在，不判断现有脚本如何实现。
- 不生成脚本代码。
- 不构建脚本生成上下文资料包。

如果用户没有提供测试用例路径或测试用例文本，先要求用户补充输入，不要扫描工程目录寻找文件。

## 固定输出目录

检查结果必须输出到当前工作目录下的固定目录：

`testcase-workflow/script-clarify/`

该目录由本 Skill 管理，目录内必须包含标记文件：

`.managed-by-autotest-check-case`

每次执行本 Skill 时，必须先清空 `testcase-workflow/script-clarify/` 再重新生成。清空前必须确认目录内存在 `.managed-by-autotest-check-case`；如果目录不存在，直接创建；如果目录存在但没有该标记文件，不要清空，先向用户说明并等待确认。

该目录中的结果只代表“测试用例文本层检查”，不是工程落地可行性结论。

## 核心原则

### 只看用例文本

本 Skill 的判断来源只能是用户提供的测试用例内容及同一用例文档内的说明。允许指出“后续自动化落地分析需要验证页面物料/元素定位/工程实现”，但不得在本阶段读取或判断这些材料。

### 不替用户补规则

发现用例描述不完整、不一致或无法判断时，必须标注问题并提问。不允许：

- 自行补造需求未写清的业务规则。
- 自行猜测未写明的断言文案。
- 自行推断未写明的测试数据、账号、角色、权限或平台差异。
- 自行决定某条用例最终是否进入脚本生成范围。

### 初判只限文本层

“文本层自动化候选初判”只回答：从用例文本看，该用例是否具备进入后续自动化落地分析的条件。最终是否能生成稳定脚本，由 `/autotest-summary-case` 在读取页面物料和工程上下文后判断。

## 工作流程（4 步，必须按顺序执行）

### 步骤 1：读取用户提供的测试用例

只读取用户显式提供的测试用例输入：

1. Excel 文件：用 Python openpyxl 读取，提取用例 ID、用例名称、模块、优先级、功能入口路径、前置条件、测试数据、操作步骤、预期结果、适用平台、备注等列；如同一工作簿包含公共前置条件、测试数据说明、待确认问题，也可一并读取。
2. 文本用例：按用户粘贴的结构提取用例 ID、名称、功能入口路径、前置条件、步骤、预期结果等信息。
3. 其他用例材料：只有在用户明确提供路径或内容时才读取。

不得为了补齐上下文而主动读取工程目录中的任何文件。尤其禁止读取：

| 禁止读取的材料 | 原因 |
| --- | --- |
| 页面映射表 | 属于后续自动化落地分析，不属于用例文本规范检查 |
| 页面 UI 树/截图/prompt/crops | 属于元素定位和页面物料验证，不属于本阶段 |
| PageObject 目录 | 属于脚本设计和复用判断，不属于本阶段 |
| 已有 pytest 用例目录 | 属于脚本生成风格和复用判断，不属于本阶段 |
| 正式图片目录 `images/` | 属于图片兜底交付检查，不属于本阶段 |
| `CLAUDE.md` / `README.md` / 工程规则文件 | 属于工程实现约定，不属于本阶段 |

如果输入缺少测试用例主体，输出 `input-inventory.md` 和 `clarify-issues.md`，标记为「补充用例后继续」，并停止后续审查。

### 步骤 2：逐条审查测试用例规范性

对每条测试用例，从以下维度审查：

| 维度 | 审查内容 | 发现问题时的处理 |
| --- | --- | --- |
| 用例 ID | 是否唯一、稳定、可追踪 | 重复/缺失 -> 缺失或矛盾 + 阻塞 |
| 用例名称 | 是否表达一个明确验证目标 | 过泛/多个目标混杂 -> 含糊或歧义 |
| 模块/功能点 | 是否能看出所属业务模块和功能点 | 缺失 -> 缺失 + 待补充 |
| 功能入口路径 | 入口路径是否完整（从首页到操作页面） | 缺失/跳步/入口不唯一 -> 缺失、含糊或阻塞 |
| 前置条件 | 是否明确账号、角色、权限、数据、设备/平台状态 | 不明确 -> 含糊或缺失 |
| 测试数据 | 是否具体、稳定、可准备、可恢复；数据准备成本是否与自动化收益匹配 | 不具体/不可控 -> 风险或阻塞；高成本且收益不高 -> 不适合自动化候选 |
| 操作步骤 | 必须是体现在UI层面可执行/观测的操作，是否顺序清晰、可复现、无跳步、无隐含动作 | 跳步/动作含糊 -> 缺失或含糊 |
| 预期结果 | 是否具体、UI层面可观察、可断言 | 只写“成功/正常”等 -> 含糊；不可观察 -> 不适合自动化候选 |
| UI 尺寸/视觉规格 | 是否主要验证高度、宽度、间距、字号、坐标、pt/px、视觉对齐等 UI 尺寸或视觉样式 | 涉及 UI 尺寸/视觉规格 -> 建议保留手工或视觉专项测试 |
| 平台差异 | iOS/Android/Web/后台等差异是否说明 | 差异影响步骤或预期但未说明 -> 含糊 |
| 账号/角色差异 | 多账号、被分享人、管理员等角色是否明确 | 未说明 -> 含糊或阻塞 |
| 数据清理 | 用例是否会新增、修改、删除数据，是否说明恢复方式 | 未说明 -> 风险或待补充 |
| 独立性 | 是否依赖前一条用例结果或不可控外部状态 | 强依赖 -> 风险或不适合自动化候选 |
| 自动化候选性 | 仅从文本看，是否可复现、可观察、可断言、数据可准备，且数据准备成本与收益匹配 | 不满足或高成本低收益 -> 建议保留手工或专项测试 |

功能入口路径只做文本层检查：判断用例是否写明“从哪里进入功能”和“如何到达操作页面”。不得在本步骤中检查入口路径对应的页面是否有映射、元素是否可定位、UI 树是否存在、图片是否齐全、PageObject 是否存在。

数据准备成本必须显式评估：

| 成本等级 | 文本层判断标准 | 默认处理 |
| --- | --- | --- |
| 低 | 单账号、少量稳定业务数据，可通过用例步骤或常规前置准备 | 可进入自动化落地分析 |
| 中 | 需要特定录音、标签、关键词、状态组合，但数量少、可稳定复用、可清理 | 需确认数据方案后进入 |
| 高 | 需要大量数据、复杂排序/分页数据、多账号权限关系、服务端索引刷新、跨状态链路或难清理数据 | 若收益不高，判为「建议保留手工或专项测试」 |
| 很高 | 需要跨端/跨系统操作、网络/系统状态控制、新老版本环境矩阵、硬件或外部环境配合 | 默认判为「建议保留手工或专项测试」 |

若数据准备成本为「高」或「很高」，且该用例不是高价值核心回归或高频故障场景，必须判为「建议保留手工或专项测试」，不得列为自动化候选。若用户明确说明该用例收益高或已有低成本造数能力，再降级为「需修订/确认后进入」并要求说明数据方案。

### 步骤 3：输出文本层自动化候选初判和问题清单

将审查结果写入固定输出目录 `testcase-workflow/script-clarify/`。

每条用例必须输出“文本层自动化候选初判”。初判只代表用例文本质量，不代表工程可实现性。

| 初判结论 | 判定条件 | 下一步 |
| --- | --- | --- |
| 可进入自动化落地分析 | 用例文本清晰，功能入口路径、前置/步骤/预期/数据足够明确，预期可观察，文本层无阻塞问题 | 交给 `/autotest-summary-case` 做页面物料和工程可行性分析 |
| 需修订/确认后进入 | 存在文案、规则、步骤、数据、平台、角色等含糊点，但用户确认后可继续 | 写入 `clarify-issues.md`，等待用户确认或修订 |
| 建议保留手工或专项测试 | 预期无法客观观察，涉及 UI 尺寸/视觉规格，依赖硬件/外部系统/跨账号人工协作/不可控状态，数据准备成本高且收益不高，或更适合人工/API/专项验证 | 保留为手工用例或转专项测试 |
| 信息不足，无法判断 | 缺少核心字段，无法判断步骤、数据或预期 | 补充用例内容后重新检查 |

初判必须保留原因，不允许只写结论。每条用例至少记录：

| 字段 | 内容 |
| --- | --- |
| 用例ID | 测试用例编号；缺失则写“缺失”并生成问题 |
| 用例名称 | 测试用例标题；缺失则写“缺失”并生成问题 |
| 文本层自动化候选初判 | 上述四类之一 |
| 关键依据 | 支撑该初判的 1~3 条用例文本事实 |
| 阻塞项 | 没有则写“无” |
| 待确认项 | 没有则写“无” |
| 建议下一步 | 进入自动化落地分析 / 等待确认或修订 / 保留手工或专项测试 / 补充用例内容 |

问题类型标签：

| 标签 | 说明 |
| --- | --- |
| 矛盾 | 同一用例或同一用例文档内描述互相冲突 |
| 含糊 | 描述无法确定唯一行为、数据、步骤或预期 |
| 缺失 | 关键 ID、标题、功能入口路径、前置、步骤、数据、预期、平台或角色缺失 |
| 不适合自动化候选 | 仅从文本看，不具备 UI 自动化候选条件，或主要验证 UI 尺寸/视觉规格 |
| 歧义 | 同一操作、结果或规则可有两种以上解读 |
| 超范围 | 用例要求验证非 UI 行为或工程内部实现细节 |

处理方式标签：

| 标签 | 说明 |
| --- | --- |
| 阻塞 | 不确认或不修订就无法进入后续自动化落地分析 |
| 假设 | 可以提出一个文本层假设，但必须等待用户授权 |
| 风险 | 可以进入后续分析，但需要在下游保留风险 |
| 待补充 | 当前信息不足，需要补充用例字段或说明 |

输出文件清单：

| 文件 | 职责 |
| --- | --- |
| `.managed-by-autotest-check-case` | 标记文件，标识目录由本 Skill 管理 |
| `input-inventory.md` | 本次读取的用例输入、来源、状态、范围和缺失项 |
| `automation-readiness.md` | 每条用例的文本层自动化候选初判、关键依据、阻塞项、待确认项和建议下一步 |
| `clarify-issues.md` | 统一问题清单，包含用例质量问题、待确认问题、假设授权请求和待补充项 |

`input-inventory.md` 必须包含：

| 字段 | 内容 |
| --- | --- |
| 输入类型 | Excel 测试用例 / 文本用例 / 用户显式提供的其他用例材料 |
| 实际路径或来源 | 用户提供路径、附件名称或粘贴文本 |
| 状态 | 已读取 / 缺失 / 部分缺失 / 不适用 |
| 覆盖范围 | Sheet 名、模块、平台、用例数量或用户指定范围 |
| 备注 | 缺失字段、读取失败原因、下一步需要补充的用例材料 |

`automation-readiness.md` 必须使用统一表格：

| 用例ID | 用例名称 | 文本层自动化候选初判 | 关键依据 | 阻塞项 | 待确认项 | 建议下一步 |
| --- | --- | --- | --- | --- | --- | --- |

`clarify-issues.md` 必须使用统一表格：

| 问题ID | 关联用例 | 问题描述 | 来源出处 | 问题类型 | 处理方式 | 建议确认人 | 需要用户回答的问题 | 可选假设 | 用户确认状态 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |

问题 ID 使用 `P1`、`P2`、`P3` 顺序编号。初次输出时，`用户确认状态` 默认为 `待确认`。

### 步骤 4：输出自检并等待用户确认

生成文件后必须自检：

- `input-inventory.md`、`automation-readiness.md`、`clarify-issues.md` 都已生成。
- 每条读取到的用例都出现在 `automation-readiness.md`。
- 所有阻塞、歧义、缺失、不适合自动化候选的问题都出现在 `clarify-issues.md`。
- 输出中没有基于工程物料或代码的判断，例如“UI 树缺失”“页面映射不存在”“PageObject 不存在”“正式图片缺失”“已有 pytest 可复用”。
- 若发现上述违规内容，必须重写结果后再交付。

输出问题清单后，向用户展示所有「阻塞」和「假设」标签的问题，等待用户确认或修订用例。

确认完成后，引导用户：

> 测试用例文本层问题确认完毕。请执行 `/autotest-summary-case` 来读取页面物料和工程上下文，进行自动化落地可行性分析与上下文交接。

## 与其他 Skill 的衔接

上游：

- `autotest-generate-cases` 输出的 Excel 是本 Skill 的主要输入。
- 用户显式提供的测试用例文本或 Excel 也是有效输入。

下游：

- 用户确认或修订用例后执行 `/autotest-summary-case`。
- 本 Skill 的输出目录 `testcase-workflow/script-clarify/` 供下游 Skill 读取。
- 下游 Skill 负责读取页面映射表、UI 树、截图、PageObject、已有 pytest 用例、正式图片目录和工程规则，并完成真正的自动化落地可行性分析。
- 下游 Skill 必须把 `automation-readiness.md` 中的“文本层自动化候选初判”作为输入，但不得把它当作最终脚本生成范围。

完整链路：

```text
autotest-check-prd -> autotest-summary-prd -> autotest-generate-cases
-> autotest-check-case -> autotest-summary-case -> autotest-generate-script
```
