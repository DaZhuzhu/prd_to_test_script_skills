## 需求质量审查

**这是写用例之前必须完成的独立步骤。**不要偷偷替产品做决定，也不要假设需求一定正确。

需求文档中不仅会有"缺失"，还会有"不准确"的问题。以下是必须检测并标识的需求质量问题类型：

| 问题类型 | 标签 | 说明 | 示例 |
|---------|------|------|------|
| 需求矛盾 | 🔴矛盾 | 同一功能在不同段落/不同页面的描述不一致 | A处说"超60秒自动保存"，B处说"不限时长" |
| 需求含糊 | 🟡含糊 | 描述无法确定唯一的预期行为 | "适当提示""合理时间""尽量减少" |
| 需求缺失 | 🔴缺失 | 关键场景/异常/边界/文案未提及 | 只写了成功流程，没写失败场景；没写空态/缺省文案 |
| 需求不可测 | 🟡不可测 | 描述无法在UI层面客观验证 | "体验流畅""加载快速""界面美观"（无量化标准） |
| 需求歧义 | 🟡歧义 | 同一句话可有两种以上解读 | "点击后返回首页"（是当前页面替换还是新开页面？） |
| 需求超范围 | ⚪超范围 | 描述了技术实现细节而非用户行为 | "调用xx接口写入xx数据库表"（用户不关心实现） |

### 审查输出格式

```markdown
## 需求质量问题清单
| 问题 | 关联功能点 | 问题类型 | 影响范围 | 原文出处 | 建议确认人 | 处理建议 |
| "超60秒自动保存"与"不限时长"矛盾 | 内容保存 | 🔴矛盾 | 影响边界用例和自动化断言 | §3.2 vs §5.1 | 产品 | 确认是否限时长，统一描述 |
| "适当提示"无具体文案 | 操作失败 | 🟡含糊 | 影响异常用例的预期结果 | §4.3 | 产品 | 补充精确提示文案 |
| 未提及无网络时的表现 | 数据导出 | 🔴缺失 | 影响网络异常用例 | 全文 | 产品 | 补充网络异常处理逻辑 |
| "体验流畅"无量化标准 | 页面加载 | 🟡不可测 | 无法编写可执行的预期结果 | §2.1 | 产品 | 明确加载时间标准（如<3s） |
| "点击后返回首页" | 操作完成 | 🟡歧义 | 影响用例的页面流转预期 | §3.4 | 产品 | 确认是替换还是新开 |
| "调用xx接口写入xx表" | 数据提交 | ⚪超范围 | 不影响用例编写，但不应出现在需求中 | §6.2 | 开发 | 从需求中移除实现细节 |
```

### 审查原则

- **逐段比对**：阅读需求文档时，注意同一功能在不同位置（概述、详细设计、交互设计、附录等）的描述是否一致。
- **逐项追问**：对每个功能点追问：成功怎么表现？失败怎么表现？边界怎么表现？不同角色怎么表现？不同状态怎么表现？不同网络怎么表现？新服务端 vs 旧服务端表现分别是什么？新客户端 vs 旧客户端表现分别是什么？如果需求没有回答，就标记为缺失。
- **文案精确性**：对每个弹窗/Toast/标签/按钮文案追问：需求有没有给出精确文案？如果没有，标记为含糊或缺失。
- **可测性**：对每条预期结果追问：测试人员能不能通过看UI、查数据、听声音等方式客观判断通过/不通过？如果不能，标记为不可测。

## 需求发散检查

需求发散检查用于发现需求文档没有直接展开、但可能影响后续测试用例生成的补充确认点。发散结果不是已确认需求，不能直接当作产品行为写入定稿需求。

### 发散方向

- 角色和权限：是否存在不同角色、无权限、越权访问场景。
- 状态组合：是否存在不同前置状态、状态矩阵、空态、禁用态、处理中、失败态。
- 输入边界：是否存在空值、最小值、最大值、超长、非法字符、前后空格等边界。
- 异常流程：是否存在取消、返回、关闭弹窗、重复操作、失败重试、外部依赖断开。
- 网络状态：是否涉及无网络、弱网、网络恢复、请求超时。
- 前后台/进程状态：是否涉及切后台、锁屏、杀进程、浏览器刷新、会话过期。
- 版本兼容：是否涉及新旧客户端、新旧服务端、不同平台或不同硬件/固件版本。
- 数据一致性：是否涉及修改后多页面、多端、列表与详情同步。
- 并发/竞态：是否涉及快速连续点击、多人/多端同时操作同一数据。

### 发散输出格式

```markdown
## 需求发散问题
| 发散场景 | 关联功能点 | 为什么需要确认 | 原文依据 | 建议确认人 | 处理方式标签 |
| 无网络提交后的提示和恢复方式 | 数据提交 | 需求涉及提交行为，但未说明网络异常表现，影响异常用例和预期结果 | §3.2 | 产品 | 🟡风险 |
| 删除操作是否需要覆盖取消路径 | 删除 | 需求写了删除成功，但未说明取消/关闭弹窗后的表现 | §4.1 | 产品 | ⚪待补充 |
| 多端同时修改同一数据的冲突处理 | 名称修改 | 需求涉及数据修改和同步，但未说明并发冲突处理 | §5.3 | 产品/开发 | 🟡风险 |
```

### 发散原则

- **必须有原文依据**：发散场景必须能关联到需求中的功能点或行为，不要凭空扩展产品范围。
- **只提出问题，不替产品决定**：如果需求没写清楚，只输出待确认问题或风险。
- **不生成测试用例**：本 Skill 只输出评审和发散问题，测试用例生成属于 `autotest-generate-cases`。
- **不污染定稿需求**：发散问题只有在用户确认后，才能由 `autotest-summary-prd` 写入 `finalized-requirement.md`。

## 处理需求不明确

不要偷偷替产品做决定。

### 处理方式标签

- 🔴 阻塞：不确认就无法生成可靠用例。
- 🟡 假设：可以先按某个假设写用例，但必须写明假设。
- 🟡 风险：能写用例，但覆盖可能不完整。
- ⚪ 待补充：当前不阻塞，但执行或自动化前需要补充。

### 需求质量问题标签

- 🔴 矛盾：同一功能在不同位置描述不一致。
- 🟡 含糊：描述无法确定唯一的预期行为。
- 🔴 缺失：关键场景/异常/边界/文案未提及。
- 🟡 不可测：描述无法在UI层面客观验证。
- 🟡 歧义：同一句话可有两种以上解读。
- ⚪ 超范围：描述了技术实现细节而非用户行为。

**两种标签可以组合使用**。例如"缺失 + 阻塞"表示需求没有提到该场景，且不确认就无法写用例。

### 输出格式

```markdown
## 待确认问题
| 问题 | 关联功能点 | 影响 | 原文出处 | 建议确认人 | 问题类型标签 | 处理方式标签 |
| 无数据时，点击导出展示什么提示？ | 数据导出 | 影响异常用例和自动化断言 | 全文 | 产品 | 🔴缺失 | 🔴阻塞 |
| 时间格式统一为yyyy-mm-dd | 全局展示 | 影响所有时间展示用例断言 | §3.1 | 产品 | 🟡含糊 | 🟡假设 |
| "超60秒自动保存"与"不限时长"描述矛盾 | 内容保存 | 影响边界用例 | §3.2 vs §5.1 | 产品 | 🔴矛盾 | 🔴阻塞 |
| "适当提示"无具体文案 | 操作失败 | 影响异常用例的预期结果 | §4.3 | 产品 | 🟡含糊 | 🟡假设 |
| "体验流畅"无量化标准 | 页面加载 | 无法编写可执行的预期结果 | §2.1 | 产品 | 🟡不可测 | 🟡风险 |
```

## PRD 输入格式适配

本 Skill 支持以下格式的 PRD 输入，每种格式的处理方式不同：

| 格式 | 处理方式 | 说明 |
|------|---------|------|
| **Markdown (.md)** | 直接读取 | 最理想的输入格式，结构清晰，易于提取功能点、表格、文案 |
| **PDF** | 需先提取为文本 | 使用 Python 的 `pdfplumber` 或 `PyPDF2` 提取全文；如果 PDF 包含复杂表格或图片，提取可能不完整，需标注"原文可能丢失格式信息" |
| **Word (.docx)** | 需先提取为文本 | 使用 Python 的 `python-docx` 提取段落和表格；表格数据可能需要手动校验 |
| **飞书/钉钉文档** | 用户粘贴文本 | 用户从在线文档复制文本直接粘贴到对话中；格式可能丢失，需标注"粘贴格式可能与原文有差异" |
| **口述需求整理稿** | 用户直接提供 | 由用户自行整理的结构化文本；格式不规范时需标注"非标准需求文档格式" |

### PDF 提取步骤（当输入为 PDF 时）

如果用户提供的是 PDF 文件路径，按以下步骤处理：

1. 使用 Python 脚本将 PDF 提取为纯文本：
   ```python
   # 仅供提取文本参考，不是测试用例的内容
   import pdfplumber
   with pdfplumber.open("需求文档.pdf") as pdf:
       text = "\n".join([page.extract_text() for page in pdf.pages])
   ```
2. 将提取的文本作为需求输入，开始正常工作流程。
